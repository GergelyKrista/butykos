1. Why This Doc Exists

As of 2026-05-07, the project just pivoted from Alcohol Empire Tycoon (single-player, Phase 7B Save/Load complete, ~7 facilities and 13 machines wired into a working factory-interior loop) to Drinkustry — a four-player asymmetric co-op cyberpunk megacorp game.

The three design docs in this folder describe the destination. They do not describe the engine retrofit. This doc fills that gap. It pins architectural decisions before Phase 8 work starts so refactor work doesn't have to be redone every time a new system lands on top of an old assumption.

Frame
The Phase 7B build is the foundation, not legacy. The autoload-singleton structure, the data-driven JSON content pipeline, the dual-layer (isometric strategic + orthogonal interior) split, the EventBus signal hub, and the working factory-interior prototype are load-bearing assets. The pivot does not throw them out; it adds an ownership layer, a multi-player concept, and four new content domains on top.

This doc is decisions, not exhaustive specs. Each section names the choice, names the alternative that lost, and names the reason. Detailed schemas, signal lists, and migration scripts are deliberately out of scope — those land in Phase-8-onward implementation tickets, with this doc as their constraint.

Document references throughout: CLAUDE.md for the current code state, the 2026-04-30 / 05-01 / 05-02 design docs for game design intent. Where this doc and CLAUDE.md conflict, this doc wins forward — CLAUDE.md describes the past tense build, this doc describes the constraint surface for the next one.

2. Architectural Principles

The rules every later section must follow. If a recommendation in §3–§7 violates one of these, the recommendation is wrong, not the principle.

P1 — Additive over rewrite
Singletons stay. Ownership becomes a layer on top, not a replacement for any existing manager. Anything that already works in Phase 7B (factory interior, vehicle pathfinding, road grid, save/load) keeps working through the pivot.
P2 — Theme-agnostic core
Cyberpunk is reskin. Every mechanic name, signal, JSON key, and schema field stays neutral (integrity_meter, not chrome_loyalty; utility_network, not neon_grid). Theme lives in display strings, sprites, audio. If we ever pivot the theme again the engine survives intact.
P3 — Data-driven everywhere new
Events, recipes, corps, utilities, contracts, demand models — all in JSON under data/. The Phase 7B precedent (facilities, machines, products, research, roads all as JSON) is the standard. New code defines schema; new content does not require code changes.
P4 — Hot-seat first, networked last
Phase 8 ships as four corps on one machine, taking turns or playing in parallel via a single shared simulation. Networking lands at Phase 12. But every decision made now must hold at the network boundary. If a design works hot-seat but breaks at MP, it is wrong now, not later.
P5 — One catchment rule, applied everywhere it fits
Per the 2026-05-02 doc's Recommendation: pick one catchment-radius behavior and apply it identically to Agri silos, Industrial output bays, and Economic markets. Owned by Logistics conceptually — see §6.
P6 — Per-corp data namespaced from day one, even pre-MP
Even in single-corp dev playthroughs, every owned entity carries a corp_id. There is no "no-corp" or "default corp" except the explicit "shared" sentinel. This avoids a future refactor where every system has to learn ownership at once.

3. Ownership / Permission Layer

The keystone change. Every singleton currently assumes one player. None of the manager files (core/, systems/) contain the words corp, player_id, or owner. That has to be the first thing that changes — every later subsystem decision depends on ownership being present.

3.1 Where ownership lives

SET Add a corp_id: String field to every owned entity:

  • Facility (WorldManager.facilities) — set at place_facility() from the active corp.
  • Machine (FactoryManager.factory_interiors[fid].machines) — inherits from parent facility.
  • Connection / Route (LogisticsManager.connections) — set by route creator; cross-corp routes require both corps' consent (deferred — see Q-A1).
  • Vehicle (LogisticsManager.vehicles) — inherits from parent connection.
  • Contract (MarketManager.active_contracts) — set when contract is offered; only owner can accept.
  • Research node unlock state (ResearchManager.unlocked_techs) — partition by corp_id; shared tier uses "shared".
  • Road tile (WorldManager.road_grid) — unowned by default ("shared"); see §3.4.
  • Field-to-farmhouse links (WorldManager.field_parents, farmhouse_children) — fields inherit corp from farmhouse. Existing farmhouse logic survives unchanged.

Active player indicator on GameManager: 

var active_corp_id: String = "agri"  # or "industrial", "logistics", "business"
var local_corp_id: String = "agri"   # in MP, the corp this client controls; equals active in hot-seat

3.2 Validation pattern

SET Each manager exposes a can_<action>(corp_id, ...) predicate that returns {allowed: bool, reason: String}. UI calls the predicate first; manager re-checks before the mutation as defense-in-depth. Managers do not silently filter. An invalid action returns a reason, never a no-op. This matters now (clean error UX) and even more later (network desync surfaces immediately as a rejection rather than silent state divergence).

Existing patterns to mirror: WorldManager.can_place_facility(), DataManager.validate_facility_requirements(). Both already return {can_build, reason}-shaped dictionaries. Extend that convention; do not invent a new one.

3.3 The "shared" sentinel

Reserve corp_id: "shared" for entities no single corp owns:

  • Roads (CS2-style — anyone can drive on them, anyone can build them, removal requires consent or vote — see Q-A1).
  • Shared-tier research nodes.
  • Neutral utility infrastructure (water mains, power trunk lines if Logistics builds them as public goods).
  • NPC-emitted contracts before acceptance.

World tiles default to unowned, not "shared". Unowned means no corp claim; "shared" means explicitly all corps. The distinction matters for permission queries.

3.4 Migration path

LOCKED Save schema break. Legacy Phase 7B saves load with all owned entities defaulting to corp_id: "single". A migration script promotes "single" to one of four corps based on facility type:

  • Fields, farms, water sources → "agri"
  • Mills, breweries, distilleries, packaging plants → "industrial"
  • Storage, depots, road networks → "logistics"
  • Markets, taverns, trade offices → "business"

Migration runs once, in SaveManager._apply_save_data(), when version < 3. No reverse migration. Pre-pivot saves are convertible once; post-pivot saves are not downgradable.

Save schema break is non-negotiable
The current CURRENT_SAVE_VERSION = 1 in core/save_manager.gd. Bumping to v3 (skipping v2 to leave headroom for an interim shape) is a hard break — old saves load via migration, but the file format changes shape. Communicate this in patch notes.

4. Multiplayer Architecture — Decided Now

Networking is Phase 12. Every line of code from Phase 8 onward is written under the model chosen below. Pinning it now avoids the death-march scenario where Phase 12 discovers half the systems are unshippable to MP and need rewriting.

4.1 The recommendation

Authoritative-host + deterministic action pipe + lockstep on tick boundaries
Host runs the simulation. All state mutations across all managers go through a single pipe: submit_action(corp_id, action_type, payload). Clients submit actions; host validates, sequences, broadcasts. Simulation advances on fixed game-tick boundaries — actions queued during a tick fire in deterministic order at the next tick.

Hot-seat is the same pipe with the host being the local machine and all four corps submitting from the same client. The network code is just a transport swap on top of an architecture that already exists locally.

4.2 What this means for Phase 8 code

  • EventBus becomes the local notification layer. It still does what it does — UI listens to facility_placed, money_changed, etc. — but it is no longer where mutations originate. Mutations originate in submit_action(); EventBus emits as a consequence of the manager applying the validated action.
  • Every WorldManager.place_facility() / FactoryManager.place_machine() / LogisticsManager.create_connection() call from UI gets routed through the action pipe. The managers themselves keep their current API as the underlying mutator — the pipe is the wrapper.
  • Determinism becomes a hard constraint (see §9).

4.3 Why this beats alternatives

ModelBandwidthReplay/saveDisconnect handlingVerdict
Authoritative state-sync (server pushes full state diffs)High — every facility/machine/inventory change crosses wireBad — state snapshots, not action logEasy — late-joiner just downloads stateRejected — bandwidth scales with sim, not player input
Lockstep peer-to-peer (no host)LowGood — action log is the saveHard — disconnect is a stall; desync is unrecoverableRejected — disconnect recovery is the killer for a 4-player co-op
Authoritative-host lockstep (chosen)Low — actions onlyExcellent — action log + tick number = perfect replayTractable — host arbitrates, late-joiner gets state snapshot then catches up via action logSelected

4.4 The action pipe shape

# core/game_manager.gd (Phase 8 addition)
func submit_action(corp_id: String, action_type: String, payload: Dictionary) -> Dictionary:
    # 1. Validate (manager-specific can_ predicate)
    # 2. Sequence (assign monotonic action_id, attach tick number)
    # 3. Apply (call into the appropriate manager)
    # 4. EventBus emits as consequence (not as cause)
    # 5. Return {accepted: bool, reason: String, action_id: int}

LOCKED Every UI-driven mutation goes through this. No exceptions. Phase 8 starts with the skeleton wired but unenforced (managers still callable directly during transition); Phase 9 enforces it.

The two things this is not
Not a command/undo system — actions are not reversible. Not a generic event bus — actions are requests for mutation, not notifications of fact. Notifications stay on EventBus.

5. Save Schema Migration

Current save (core/save_manager.gd) is a flat dictionary keyed by subsystem (world, factories, logistics, economy, production, market, research) plus metadata. CURRENT_SAVE_VERSION = 1. Skip v2 (leave headroom) and target v3 directly.

5.1 New shape SHIPPED — step 2, 2026-05-18

The shape below reflects what actually shipped in commit 6ac4ffa, not the originally-spec'd shape. Two deviations from the first draft of this section, both load-bearing: (1) money stays in shared.money rather than splitting per corp — deferred to the EconomyManager refactor; (2) connections and vehicles live under each owning corp's partition rather than under a single logistics section, so an ownership transfer (e.g., Agri buys out a route in v1.5+) is a clean dict move. 

{
  "version": 3,
  "save_metadata": { wall_timestamp, date, ... },  // for save-list display
  "shared": {
    "money": ...,                                  // single wallet, splits per-corp at v3→v4
    "world_tiles": {...},
    "road_tiles": {...},
    "market_state": {...},                         // current_prices, demand pressure, etc.
    "research": {...}                              // shared higher-tier research (placeholder shape)
  },
  "corps": {
    "agri":       { "facilities": {...}, "machines": {...}, "connections": {...}, "vehicles": {...}, "contracts": [...], "research_internal": {...} },
    "industrial": { ... },
    "logistics":  { ... },
    "business":   { ... }
  },
  "utilities": null,    // placeholder for step 5 (UtilityManager)
  "events":    null     // placeholder for step 11 (EventManager)
}

Each corp partition holds only its owned entities. Connections/vehicles live under the owning corp (typically logistics in v1, but Agri irrigation networks land under agri). The corp_id field on each entity is the source of truth; partition location follows it.

save_metadata sits at the top level (not inside shared) because the save-slot list reads it without opening the full save body.

5.2 Migration rule (v1 → v3) SHIPPED — step 2, 2026-05-18

  1. Read version; if < 3, run migration.
  2. Backup first. Write the original to <slot>.save.v1.bak before any overwrite. One-time per slot; never overwritten.
  3. Detect intra-v1 shape: (a) pre-step-0.5 (legacy routes key, no corp_id); (b) post-step-0.5 (canonical connections, still no corp_id); (c) post-step-1 (canonical, has corp_id). All three are handled by the same migration entry point.
  4. Walk all facilities and machines. For each, if corp_id is present and not "single", trust it. Otherwise look up corp via _facility_type_to_corp() / _machine_type_to_corp() (full coverage of data/facilities.json and data/machines.json; unknowns default to industrial with push_warning).
  5. Walk connections and vehicles. Migrate to the owning corp's partition (typically logistics).
  6. Move single-wallet money into shared.money as-is. Do not split per corp here — that is the v3→v4 EconomyManager refactor (see §6 and §7 step 4.5). Splitting now would force EconomyManager scope into the schema commit and leave three corps with $0 against a UI that has no corp switcher yet.
  7. Move market state, road tiles, and world tiles into shared.
  8. Build the full v3 dict in memory; only write to disk after migration completes. Atomic write (.tmp → rename) to prevent partial-write corruption.
  9. Bump version to 3 on the in-memory dict; subsequent saves are v3-only.

Tolerant legacy-key reads (from step 0.5's TODO markers) are deleted after this migration ships; their fallback logic is fully consumed by the migration entry point.

5.3 Format decision

LOCKED Stay JSON. Human-readable wins over binary size every time during an active dev cycle, and no current save approaches a size where parse cost matters. If save size becomes a problem post-launch, gzip the JSON; do not invent a binary format.

The existing JSON.stringify(data, "\t") in SaveManager._write_save_file() stays. Indent with tabs, debuggable in any editor.

6. Subsystem-Reuse Map

For each new system implied by the design docs, the call: new manager vs extend existing. The bias is extend — every new manager is one more thing that has to learn ownership, save/load, action-pipe, and determinism. New managers only when the data shape genuinely diverges.

UtilityManager
NEW · single manager, three named graphs
Water, power, sewage are all the same propagation algorithm with different tags. One manager, three graphs (graphs["water"], graphs["power"], graphs["sewage"]), not three managers. Edges are the existing road graph extended with pipe-only nodes (CS2-style: power follows roads by default, dedicated lines optional per the 2026-05-02 logistics card). Why not extend Logistics: utility graphs are passive (no vehicles, no schedules); fundamentally different update pattern (dirty-flag propagation vs. per-tick vehicle movement).
Irrigation → LogisticsManager extension
EXTEND
Irrigation is a logistics graph, agri-flavored. Same data structure as roads + connections. Add a network_kind: "transport" | "irrigation" | "utility" tag per network. Different visual mode in scenes/world_map/route_renderer.gd; same backing structure. Why not new manager: irrigation pipes have the same place / connect / pathfind / save shape as roads. Splitting them duplicates A* and connection-graph code.
CatchmentManager
NEW · spatial-query infrastructure
Catchment radius (per-station, per-silo, per-market — see 2026-05-02 §6) replaces direct-connect routing as the default model. Lives standalone for performance — spatial-hash queries are hot-path and don't belong in any existing manager's process loop. Owned by Logistics conceptually (priority resolution when ranges overlap is Logistics' job, per the 2026-05-02 doc) but independent of LogisticsManager's connection state. Existing direct connections become a degenerate catchment (radius 0). The current connection_paths structure stays, just stops being the only routing mechanism.
OverlayManager
NEW · render-layer in world_map scene
Generic data → tile-tinted texture pipeline. Suitability, pollution, water coverage, power coverage, market reach, integrity all go through the same path. Build it generic the first time — second-overlay-onward becomes content, not code. The Phase 7B grid renderer (scenes/world_map/grid_renderer.gd) is overlay-shaped already; OverlayManager is a slim service that feeds it data sources.
DemandManager
NEW · settlement tiers + reputation
Sales become spatial: outlets read demand at their catchment, not flat. Owned conceptually by Business corp but the manager itself is system-level. Replaces auto_sell behavior — products sell at outlet-resolved prices via demand model, not at flat product_prices dictionary. Why not extend MarketManager: MarketManager is a global price-fluctuation model. Demand is per-settlement, per-product, per-tier — needs spatial state MarketManager doesn't have. They cooperate (demand modulates the price MarketManager publishes for that location) but are different shapes.
EventManager
NEW · data-driven narrative engine
Reads facts from EventBus (drought conditions, money below threshold, tech unlocked). Loads event JSON from data/events/*.json. Emits as event_fired / event_resolved signals. Generates Business contracts via EventManager → DemandManager pipe — no direct coupling between Business systems and Agri/Industrial/Logistics. Events are the broker. This keeps the cross-corp loops (per 2026-05-02 §7) data-driven and modular. Drought event triggers Agri yield drop AND a Business import contract — but neither corp's manager knows about the other; they both react to the event.
SecurityManager
NEW · integrity / espionage
Owns integrity meters per region (per 2026-05-02 Q3 — region is the strongest default). Outposts are facilities owned by Business, placed on the world map, visible to other players. Reuses placement and overlay infra; does not get its own coordinate system. Espionage actions submit through the action pipe like everything else. Why not extend MarketManager: Security has its own state shape (region meters, outpost coverage, threat queue) that has nothing to do with prices.
ResearchManager (refactor)
REFACTOR IN PLACE · two-layer tree
Add tier: "corp_internal" | "shared" and corp_id: String to research nodes. Spend logic accepts multi-source contributions for shared-tier nodes; proposal/vote UI deferred to Phase 11. Existing 40 nodes (per data/research_tree.json) migrate to tier: "corp_internal", distributed across the four corps based on the branch they touch (the existing 8 branches: agriculture, grain_processing, brewing, distillation, aging, packaging, logistics, commerce already align cleanly with the four corps). Branch-to-corp mapping is already implicit: agriculture → Agri; grain_processing/brewing/distillation/aging/packaging → Industrial; logistics → Logistics; commerce → Business. The migration writes itself.
EconomyManager
EXTEND · per-corp money
Replace single money: int with money_by_corp: Dictionary. Every transaction takes a corp_id. Maintenance collection iterates per-corp and bills the corp that owns the facility. Disabled-facilities tracking gains a per-corp partition. Smaller refactor than it sounds — every existing call site already has access to a facility id; that id resolves to a corp. Most call sites need a one-line addition.
FactoryManager
UNCHANGED · already in good shape
The factory-interior model is the prototyped depth bar (per 2026-05-02 §2). It already separates per-facility state cleanly. Adding corp_id on the parent facility is enough — interior data inherits ownership from the parent. No internal restructure. Surprising finding from code review: _next_machine_id is global on FactoryManager, not per-interior. That's correct and stays — global id space simplifies routing references.
WorldManager
EXTEND · per-tile claims, otherwise unchanged
Add per-tile claim metadata when needed (probably not v1 — facilities-on-tiles already encode ownership through the facility's corp_id). Isometric coordinate system, A* pathfinding, road grid, farmhouse-field relationships all stay. The 50×50 grid is fine for four-corp v1; if maps grow later that's a separate decision.

9. Determinism

The MP plan in §4 needs deterministic simulation. That means the same action log applied to the same state on two machines produces identical results. Phase 7B is not deterministic; the changes are localized but real.

9.1 Specific things to fix

  • randf() / randi() calls. Audit found nine sites in systems/market_manager.gd alone (price drift, trend changes, contract generation). Wrap a per-game RNG state seeded by save-load and advanced through the action pipe — every action that needs randomness draws from a deterministic stream.
  • Dictionary iteration order. Godot's GDScript dictionaries iterate in insertion order, which is deterministic within a single process but can diverge across machines if entries are inserted in different orders due to async signal processing. Where iteration order affects state mutation (rare but real — auto-dispatch sweeps in Logistics are a candidate), iterate sorted keys.
  • Time.get_unix_time_from_system(). Currently called in save_manager.gd for save metadata and (more dangerously) implicitly throughout via _process(delta) using wall-clock delta. Replace sim-relevant time with a game-tick counter advanced by the action pipe. Wall time stays only for save-list display sort.
  • Physics-based positioning. Vehicles already use fixed-tick movement (_update_vehicles(delta)) — verify under varied frame rates that integration is identical. Likely fine but unverified.

9.2 Enforcement

Q Per Q-A6: hard (assert) or soft (linter)? Recommendation: targeted asserts in the action pipe (every action either declares deterministic or routes through the seeded RNG; non-declared randf() in a manager hot path is an assert in debug builds). Plus a one-shot linter pass for the existing codebase.

Determinism is a property of the action pipe, not every line
UI animations, particle systems, sprite jitter — all the cosmetic stuff — can use randf() freely. Determinism applies to the simulation, not the presentation. The pipe is the boundary.

11. Open Architectural Questions

A1 — Is corp ownership immutable per-game, or can corps trade buildings?
Recommend immutable for v1, tradable for v1.5+. Trade introduces vote-on-removal-of-shared-roads questions and complicates ownership UX. Ship without it; revisit when the four-corp loop is proven.
A2 — Hot-seat single-machine vs multi-process from day 1?
Recommend single-machine until the action pipe is proven (steps 1–4 of §7), then multi-process for testing pre-network. Multi-process from day 1 is over-engineering — the action pipe model holds either way, but local in-process is faster to iterate on.
A3 — How does save merge across hot-seat sessions?
Recommend one save per session, no merging. All four corps live in one save file. Hot-seat means swapping active_corp_id; the save is the whole world. Avoids a merge-conflict problem nobody asked for.
A4 — Heat-map overlay texture resolution?
Recommend per-tile, GPU-accelerated, profile first. The 50×50 grid is small enough that per-tile is cheap. If it isn't, per-2×2 blocks fall back well.
A5 — Event engine triggers — pure conditional polling, or signal-driven via EventBus?
Recommend signal-driven for facts already on the bus (drought conditions emitted by climate sim, money thresholds emitted by EconomyManager), polling only for derived facts not worth bus-ifying (e.g., "agri inventory of barley exceeds X for Y ticks" — that's polling). Hybrid wins; pure-signal would balloon the EventBus surface.
A6 — Determinism enforcement — hard (assert) or soft (linter)?
Recommend linter pass + targeted asserts in the action pipe. A full determinism audit slows feature work; pragma-style assertions in the action pipe catch the cases that actually break MP. Linter as a once-a-phase sanity sweep.
A7 — Cross-corp routes: who owns a route from Agri silo to Industrial mill?
Not explicitly raised in design docs but lands here. Recommend Logistics owns transport routes by default (per 2026-05-02 Q4 — "Logistics-owned reinforces cross-corp dependency").
