1. TL;DR

  • Slice 3.3 shipped. Mash Tun, Brewer, and Bottling Line read multi-input recipes from recipes.json and produce lager end-to-end.
  • Water Pump is a new always-available world-map facility (cost $200, outputs water with no input). Lets the chain be tested without unlocking water_source.
  • Fermentation Vat removed from machines.json — Fermentation Vessel is the canonical fermenter and Vat was producing the wrong intermediate.
  • Per-product buffer caps (MACHINE_BUFFER_CAP_PER_PRODUCT = 16): machines stall when their output buffer is full. Backpressure cascades from any downstream bottleneck back through the chain.
  • State-change-only logging via _log_once: per-cycle plumbing prints are gone. Each producer / hopper / depot logs exactly one line per state transition.
  • Connection arrows in the factory interior now stop marching the instant the receiving machine fills up — visual stall indicator.
  • Shared product colors live on DataManager.PRODUCT_COLORS. Network view, factory connections, and the hopper-config popup all source from there.

2. Multi-Input Recipes (Slice 3.3)

Slice 3.1 landed the recipes.json data model. Slice 3.2 landed the per-hopper product config. Slice 3.3 is the engine work that wires them together so multi-input machines actually execute a recipe.

Recipe lookup path

A machine entry in machines.json opts in by setting a recipe_id instead of (or alongside) the legacy production block. The recipe path takes priority — if both are present, recipe_id wins. 

"mash_tun": {
    "name": "Mash Tun",
    "recipe_id": "mash_tun_lager",
    ...
}

// recipes.json
"mash_tun_lager": {
    "inputs":  [{"product": "grist", "quantity": 4}, {"product": "water", "quantity": 6}],
    "outputs": [{"product": "mash",  "quantity": 4}],
    "cycle_time": 4.0
}

Engine hooks

All three changes live in systems/production_manager.gd:

  • _on_machine_placed — reads recipe_id, pulls cycle_time from the recipe, initializes the machine timer. Falls through to the legacy production.cycle_time path if no recipe.
  • _complete_machine_recipe_cycle(...) — new function. Checks every input is present in the machine's buffer, checks every output has room (see §4), then atomically consumes inputs and produces outputs. Reports the first missing input or first full output on block.
  • _machine_needs_product(machine, product) — recognizes recipe inputs (so _try_transfer_to_adjacent routes correctly into multi-input machines).
Backwards compatibility
The legacy single-input path (production.input + production.output) is untouched. Mill, Lauter Tun, Whirlpool, Cooler, Fermentation Vessel, Maturation Tank, Filtration Unit all still run on the old path — they don't need a recipe because they're single-in/single-out.

4. Per-Machine Buffer Caps

This is the most consequential mechanical change. Before this session, machine inventories were unbounded — a Mill could produce 1,000 grist into its own buffer if nothing downstream consumed it. The visible symptom was that nothing visibly halted the chain: the logs showed a healthy "producing" cadence even when the chain was broken because the producer had infinite slack.

The design

Borrowed from Satisfactory: each machine has a per-product buffer cap. If the cap is full, the machine stops consuming inputs until the output drains. Backpressure propagates up the chain. 

const MACHINE_BUFFER_CAP_PER_PRODUCT: int = 16

func _get_machine_buffer_cap(machine_def: Dictionary) -> int:
    if bool(machine_def.get("is_storage_buffer", false)):
        return int(machine_def.get("storage_capacity", MACHINE_BUFFER_CAP_PER_PRODUCT))
    return MACHINE_BUFFER_CAP_PER_PRODUCT

Cap is per-product, not total — a Mash Tun can independently hold up to 16 grist, 16 water, and 16 mash. Storage-buffer machines (the data-defined is_storage_buffer flag) override with their own storage_capacity field (Storage Tank: 100, Large Storage Tank: 300).

Where the cap is enforced

FunctionWhat it now checks
_complete_machine_recipe_cycleBefore consuming inputs, verifies every output has room (current + quantity ≤ cap). If not, resets timer and logs once.
_complete_machine_production_cycle (legacy)Same — output cap check before consume.
_try_transfer_to_adjacentTransfer amount clamped by destination's _machine_remaining_capacity.
_process_input_hopperPush to each destination clamped by destination room. Reports "blocked" when every destination is full.
_process_storage_bufferSame clamp on destination room.
Output Depots are uncapped on the facility side
Output Depots push to facility inventory, which is unbounded — trucks ship from there to other facilities. We did not put a cap on facility inventory in this slice. If you want a facility-level storage cap later, the natural place is _add_to_inventory in ProductionManager.

How backpressure cascades

The lager chain stress test (final log this session):

  1. Player adds the hops route last. Brewer has wort but no hops.
  2. Brewer needs 2 hops (has 0) — Brewer blocks on input.
  3. Lauter Tun keeps producing wort. Brewer's wort buffer fills to 16.
  4. Lauter Tun output buffer full (wort: 16/16) — Lauter Tun stalls.
  5. Mash Tun keeps producing mash. Lauter Tun's mash buffer fills to 16.
  6. Mash Tun output buffer full (mash: 16/16) — Mash Tun stalls.
  7. Mill output buffer full (grist: 16/16) — Mill stalls. Hopper backs up.
  8. Hopper blocked: malt (every destination full) — facility-side malt accumulates.

The instant hops arrive, the stack unwinds in reverse — each machine resumes as soon as its output has room and its inputs are present. The visible log narrates the cascade exactly once per transition.

5. State-Change-Only Logging

Old behavior: every production cycle printed something. A blocked Mash Tun spammed needs 4 grist every 4 seconds. A working Hopper spammed 10 malt (facility → machine_9) every 2 seconds. The console was unreadable.

The pattern

var _last_log_message: Dictionary = {}

func _log_once(state_key: String, message: String) -> void:
    if _last_log_message.get(state_key, "") != message:
        _last_log_message[state_key] = message
        print(message)

Each machine / hopper / depot has a stable state_key (e.g. "machine:facility_7:machine_9"). The message string IS the state — when the message changes, we print and update the cache. When it stays the same, we stay quiet.

Observable states

Producing
Machine running
Machine producing: Mill (4 grist every 3.0s) — fires once when the machine transitions from blocked/idle into a working cycle.
Blocked
Input shortage
Machine blocked: Brewer needs 2 hops (has 0) for recipe 'Boiling Wort with Hops' — fires once per shortage reason. Reports the first missing input.
Blocked
Output buffer full
Machine blocked: Lauter Tun output buffer full (wort: 16/16) — the backpressure indicator.
Flowing
Hopper transferring
Hopper flowing: malt (facility → machines) — at least one destination received product this tick.
Waiting
Hopper has no stock
Hopper waiting: bottles (facility stock empty) — facility inventory empty, hopper has nothing to push.
Blocked
Hopper destinations full
Hopper blocked: water (every destination full) — facility has the product but every connected machine's buffer is full.

Each transition prints once. If a Mash Tun oscillates between blocked: needs grist and producing, both lines fire on each transition — which is the right signal: state is actually changing.

6. Connection Arrows & Shared Product Colors

Shared color palette

Per-product colors moved to core/data_manager.gd as PRODUCT_COLORS + get_product_color(product_id) -> Color. Single source of truth. network_view.gd's _socket_color_for now delegates here instead of holding its own copy.

The palette is curated for slice-1 + slice-3 products (barley golden, hops vibrant green, water blue, lager warm gold, grist/wort/boiled_wort/cleaned_wort/cooled_wort tinted along the brewing pipeline). Unknown products fall back to a hash-derived hue so future content doesn't crash here.

Factory interior connection arrows

Replaced the old static "light blue line + single orange arrow" with:

  • Line color = product color (from DataManager.get_product_color).
  • Flowing = marching arrows along the line, paced by FLOW_ARROW_SPEED / FLOW_ARROW_SPACING (mirrors network view).
  • Idle = single mid-line arrow at t=0.7 (direction hint).

"Flowing" is computed by _is_connection_flowing(source_id, dest_id, product) in scenes/factory_interior/factory_interior.gd. It requires BOTH (a) the source has the product, AND (b) the destination has buffer room. So when a downstream machine fills up, arrows stop marching at exactly that connection — the player can see where the stall is.

Hopper config popup

Right-clicking an Input Hopper or Output Depot opens a product picker. Each entry now has a generated 16×16 color swatch icon matching the connection color, so the picker visibly maps to the line color the hopper will drive.

8. Open Items / Known Limits

  • Facility inventory still unbounded. Trucks ship from facility inventory, so it acts as a reservoir. If we want facility-level caps (e.g., a brewery can only hold so much malt at a time), the entry point is _add_to_inventory in ProductionManager — but that interacts with logistics dispatch and farmhouse routing, so it's a real design decision, not a one-liner.
  • _try_transfer_to_adjacent still uses the half-rule. When a producer transfers to a downstream machine, it transfers max(1, available / 2), capped by destination room. With caps in place this means producers never dump huge batches on transfer (max ≈ 16), but the half-rule itself is heuristic. Worth revisiting if balance feels off.
  • Storage tanks aren't fed by upstream machine output. _machine_needs_product returns false for non-recipe / non-production machines, so a Mill cannot directly fill a Storage Tank via a connection. Storage tanks are reachable only via Input Hopper today. If we want machine→storage connections to work, add a special-case for is_storage_buffer in _machine_needs_product.
  • Buffer cap of 16 is a guess. Validated on the lager chain — chains balance, severed connections stall in <1 min. Hasn't been stress-tested on longer or branched chains.
  • No save-game compatibility for old saves with Fermentation Vat. If a save references a machine type that no longer exists, get_machine_data returns {} and the machine becomes inert. We should handle this in the save migration path if any test saves matter.
  • Connection rendering still recreates Line2D / Polygon2D children every frame. Works fine for ~20 connections; will want to convert to a _draw() override on the connections renderer if a brewery interior gets significantly denser.
  • Branch state. Both new commits landed on testing/farmhouse+corp-switcher directly. The feature branch feature/brewery-lager-chain doesn't have the buffer-cap work because _complete_production_cycle + _on_facility_removed on testing reference farmhouse-only state (field_crop_types, _field_idle_reason, _complete_farm_field_cycle) that doesn't exist on the feature branch. Re-applying to feature requires the farmhouse work to land there first.
